> Output of a 10-agent design workflow: parallel audits -> 3 competing directions -> 3-lens judge panel.
> Winner: **Native Chameleon — "Shipped by X"** (tally: {"Native Chameleon — \"Shipped by X\"":2,"Roped In":1}).
> Final deliverables: [redesign brief](2026-06-11-redesign-brief.md) · [product story](2026-06-11-product-story.md)

## UI/UX audit

Lasso's UI is structurally sound under the hood (headless picker state, shadow DOM isolation, canonical keymap) but the surface layer is an undesigned pile of bottom-centered fixed elements with near-zero information density, no loading/progress/error feedback, invisible feature discoverability (select mode literally renders nothing), OS-based dark mode that ignores X's own theme, and several a11y gaps (no focus trap, Escape only works while the input is focused, 3%-lightness active-row highlight, white-on-X-blue text failing AA). The 'swift' claim is contradicted by a forced network fetch blocking every picker open with no spinner, and by clearing the selection before the assign calls resolve, making failures unrecoverable. Findings below are ordered roughly by product impact; the picker's positioning/anchoring, the feedback-state vacuum, and discoverability are the three redesign pillars.

- **[high] ListPicker floats detached at bottom-center, pinned by its bottom edge, so the input jumps while you type** (Layout / positioning)
  src/content/app.tsx:81 wraps the picker in `fixed bottom-24 left-1/2 -translate-x-1/2`. It is anchored to nothing — not to the focused tweet (Alt+l), not to the 'Add to list' button, not centered like a command palette. Worse, because the BOTTOM edge is pinned and the search input is at the TOP of the panel (ListPicker.tsx:47), every keystroke that changes the result count changes the panel height and physically moves the input under the user's caret. Either pin the input edge (anchor by top, or put the input at the bottom), give the panel a fixed height, or move to a centered palette at ~30% viewport height with a fixed-size scroll area. Also decide an anchoring story: keyboard invocation near the focused tweet vs. palette-center are both defensible; bottom-of-screen above two other floating pills is not.
- **[high] No loading state: Alt+l does nothing on screen until a forced network fetch completes** (Feedback states)
  src/content/app.tsx:52-59 — openPicker() awaits `listCache.lists({ force: true })` before setting any state. Between pressing Alt+l (or clicking 'Add to list') and the picker appearing there is zero UI: no skeleton, no spinner, no button pressed-state. `force: true` guarantees every single open pays full network latency, defeating the cache and the 'swift' philosophy in the same line. Open the picker shell instantly with cached lists (or a skeleton), revalidate in the background, and reconcile. This is the single biggest perceived-speed bug in the product.
- **[high] Errors collapse into a lying empty state: logged-out or rate-limited users see 'No matching lists'** (Feedback states / microcopy)
  src/content/app.tsx:56-58 catches fetch failure and calls `setLists([])`; ListPicker.tsx:75-77 then renders 'No matching lists'. Three distinct situations — fetch failed, rate-limited, user genuinely has no lists — all render the same misleading string with no retry, no error styling, no 'create a list' path. Model picker state as idle/loading/error/empty/ready and give each a designed view: error gets a retry button and a reason; true-empty gets a 'Create list' CTA (the product currently dead-ends a brand-new user completely).
- **[high] The keyboard layer is undiscoverable; the only user guidance ships in the devtools console** (Discoverability / onboarding)
  There is no onboarding, no hint UI, no shortcut legend anywhere in the rendered product. The single piece of user guidance is `console.warn('[Lasso] hover a tweet (or press j to focus one) first')` in src/content/main.tsx:107 — invisible to 100% of users. Pressing Alt+l with no target is a silent no-op. Fixes: a kbd footer row in the picker ('↑↓ navigate · ↵ add · esc'), kbd badges on ActionBar buttons ('Add to list  L'), a one-time first-run coachmark over the first tweet overlay, a '?' shortcut cheatsheet, and a visible inline nudge (small toast) instead of the console.warn when a command has no target.
- **[high] Select mode ('s') is a visual no-op — the signal is toggled and nothing in the UI subscribes to it** (Feature completeness)
  src/content/main.tsx:101-103 toggles `selection.setSelectMode(...)`, but no component reads `selection.selectMode`: overlays are injected unconditionally for every tweet (main.tsx:137-139), TweetOverlay.tsx never receives the mode, and there is no mode indicator anywhere. A user pressing 's' gets literally zero feedback that anything happened. Either delete the mode or implement it: overlays hidden by default, revealed in select mode, plus a persistent mode chip ('Select mode — x to select, s to exit') so state is always visible.
- **[high] Picker rows are bare strings — no member counts, no privacy icon, no avatar, no match highlighting, no 'already added' state** (Visual hierarchy / information density)
  src/ui/ListPicker.tsx:58-74 renders only `{list.name}` per row. Zero metadata: no member count, no private-list lock glyph, no list banner/avatar, no highlight of the fuzzy-matched characters, no checkmark when the selected author is already in that list, and no visual cue that rows are usage-ranked (listUsage is wired in app.tsx:55 but invisible — a 'Recent' section header would make the ranking legible). This is why it reads as a default <select> in a rounded box rather than a designed product. Raptive fix: two-line rows (name + 'Private · 142 members'), matched-substring emphasis, and a right-aligned check for membership.
- **[high] Assignment runs with no progress UI and clears the selection before results return — failures are unrecoverable** (Feedback states)
  src/content/app.tsx:67-75 — pick() closes the picker, immediately calls `selection.clear()`, then awaits N sequential network calls with nothing on screen. For 10 authors the user stares at a blank timeline for seconds, and if the run fails or rate-limits midway the selection is already gone, so there is no retry. Keep the ActionBar visible in an 'Adding 3 to Design Folks…' progress state, clear selection only on success, and offer 'Retry failed' on the result toast for partial failures.
- **[high] Dialog has no focus trap, no outside-click dismiss, and Escape only works while the input is focused** (A11y / keyboard)
  src/ui/ListPicker.tsx — keydown handling (including Escape, line 35-38) lives on the <input> only; if focus ever leaves it (Tab, or a click anywhere), the dialog becomes keyboard-undismissable. There is no backdrop or outside-click handler in app.tsx, no focus trap, no focus restoration to the previously focused tweet on close, and `autofocus` on a dynamically rendered shadow-DOM input is fragile across browsers. The ARIA wiring is also incomplete: role=dialog without aria-modal, input not marked role=combobox/aria-expanded/aria-controls, listbox options lack ids so aria-activedescendant can't work. Use the standard combobox pattern, a document-level Escape handler while open, and explicit .focus() on mount.
- **[high] Keyboard-active picker row is never scrolled into view** (A11y / keyboard)
  src/ui/picker-state.ts moveUp/moveDown only clamp an index; nothing in ListPicker.tsx calls scrollIntoView on the active option. With more than ~7 lists (max-h-[360px], ListPicker.tsx:45), arrowing down moves the highlight off-screen and the user navigates blind — fatal for a keyboard-first palette. Add an effect that scrolls the active option into view (block: 'nearest') on activeIndex change.
- **[high] Theme follows the OS, not X's own theme setting — guaranteed mismatch for Dim and cross-theme users** (Dark mode / theming)
  src/ui/styles.css:24-33 switches palettes on `prefers-color-scheme`, but X has three site themes (Default, Dim #15202b, Lights Out) chosen independently of the OS. OS-light + X Lights Out yields blinding white Lasso panels on a black site; OS-dark + X Default yields black widgets on white. There is also no Dim palette at all — dark surface is pure oklch(0 0 0), wrong on Dim. Detect X's theme (e.g. body background color or X's CSS vars) at mount and on change, and drive the token set from that, with three palettes.
- **[medium] Active-row highlight is a 3% lightness shift; rows have no hover state at all** (Visual polish / contrast)
  src/ui/ListPicker.tsx:68-70 — the only selected-row treatment is `bg-elevated` (oklch 0.97 vs the 1.0 white surface, styles.css:11) — nearly invisible on most monitors, and the active state is the load-bearing affordance of a keyboard palette. Mouse users get zero hover feedback (cursor-pointer only). Use a clearly visible active treatment (accent-tinted background or left accent bar + ink-strength text) and a distinct, lighter hover state.
- **[medium] White text on the X-blue accent fails WCAG AA for the button and the entire toast** (A11y / contrast)
  styles.css:15 sets --color-accent to oklch(0.68 0.15 237) (~#1d9bf0) with --color-accent-ink white; white on that blue is roughly 2.9:1, below the 4.5:1 AA threshold for the text-sm copy used in ActionBar.tsx:19 ('Add to list') and the whole Toast (Toast.tsx:12). X gets away with it on large bold CTAs; a 13-14px toast carrying the only record of what just happened should not. Use a darker accent for filled controls, or surface-colored toast with accent text/icon.
- **[medium] One cheerful blue pill for every outcome: failures and rate limits render identically to success, with no dismiss, no undo, and an unreliable live region** (Feedback states / microcopy)
  Toast.tsx styles all summaries as the accent pill; 'rate limit reached — stopped' and '3 failed' (result-summary.ts:36-39) appear in success styling with a lowercase clause glued on via middle dots. The toast force-expires at a hardcoded 4000ms (app.tsx:74) with no hover-pause, no dismiss, no 'Undo' or 'View list' action, and vanishes with no exit transition (entrance gets @starting-style, unmount is an instant pop). The aria-live region also mounts with its text already present, which screen readers frequently do not announce — keep a persistent live region and swap its content. Design success/partial/failure variants with an icon, a color signal, and a follow-up action.
- **[medium] Three fixed elements stacked at bottom-center with magic offsets (bottom-6 / bottom-20 / bottom-24) and no shared layout system** (Layout / composition)
  ActionBar.tsx:13 (bottom-6), Toast.tsx:12 (bottom-20), app.tsx:81 (picker at bottom-24) are independently absolutely positioned and collide by coincidence rather than composition: the toast slot sits flush under the picker, and on short viewports the picker + bar + toast eat the lower third of the screen and cover X's own compose FAB and toasts. Introduce a single bottom-center stack container (flex column, gap) inside the UI root so the pieces lay out relative to each other, and reconsider whether the picker belongs in that stack at all (see finding 1).
- **[medium] ActionBar says '3 selected' but never shows WHO — avatarUrl is captured and used nowhere in the product** (Information density / affordances)
  ActionBar.tsx:15 renders a bare count. TweetAuthor carries displayName and avatarUrl (selection-store.ts:4-13) and not a single component uses either. Selections persist invisibly as you scroll, so users lose track of who is in the set with no way to review or remove individuals. Render a facepile of overlapping avatars (+N overflow) on the bar, expandable to a removable chip list; also clarify the unit — '3 authors', since Lasso adds authors, not tweets, which is the product's most surprising semantic and currently never stated anywhere in the UI.
- **[medium] Always-on checkmark circles are prepended inside every author name row, shifting X's native layout on every tweet** (Tweet overlay / layout shift)
  main.tsx:31-40 prepends a 20px inline-flex host with margin into [data-testid='User-Name'] for every scanned tweet, permanently nudging X's author line and adding a hollow circle of chrome noise to the entire timeline whether or not the user is selecting. The ✓ is a raw text glyph (font-dependent metrics, TweetOverlay.tsx:27) rather than an SVG, and the unselected circle has no hover affordance (text is transparent, border never reacts; only active:scale). Gate the overlay on select mode or hover-reveal it, position it without displacing X's layout (e.g., absolutely over the avatar corner), swap the glyph for an inline SVG check, and add hover:border-accent + a hover check preview.
- **[medium] Alt+m mutes the hovered tweet's author with zero confirmation, feedback, or undo** (Feedback / destructive actions)
  main.tsx:122-127 — mute/block/not-interested run silently; success shows nothing, failure goes to console.error (line 133). The target is whatever tweet the mouse last passed over (mousemove listener, main.tsx:80-93), which can be stale relative to the user's mental model after scrolling. Muting the wrong person and never knowing is a real outcome today. Every quick action needs a result toast naming the target ('Muted @handle — Undo'), and mute/block deserve an undo window since X's APIs support unmute/unblock.
- **[low] ActionBar pops in/out with zero transition while the toast gets one — motion language is inconsistent** (Motion)
  ActionBar.tsx:9 returns null at count 0, so the bar mounts/unmounts instantly; the Toast has a @starting-style entrance; the picker (correctly, per the stated Emil principle) has none. The 'never animate keyboard-triggered UI' rule applies to the palette, not to a passive status bar appearing because of a mouse click on an overlay — the abrupt pop reads as a glitch, not as speed. Give the bar a ~150ms translate-y/opacity entrance and exit (the codebase already defines a strong --ease-out token in styles.css:19 that almost nothing uses), and add an exit transition to the toast.
- **[low] Copy is developer-flavored throughout: 'Filter your lists…', 'No matching lists', 'Nothing to add', 'N not allowed'** (Microcopy)
  ListPicker.tsx:51 placeholder 'Filter your lists…' describes the mechanism, not the task ('Add @handle to…' or 'Search your lists'); the dialog has no visible title or any indication of how many/which authors it will act on (aria-label only, line 44). result-summary.ts:36-39: 'N not allowed' for protected accounts hides the actual reason ('N protected accounts skipped'), and 'Nothing to add' reads like a shrug. A visible picker header ('Add 3 authors to a list') would simultaneously fix the title gap, the unit ambiguity, and the pre-flight confirmation gap.

## Storytelling audit

### Current story

THE CURRENT STORY, STAGE BY STAGE (as actually built):

1. DISCOVERY / STORE PROMISE — The manifest (src/manifest.config.ts) promises "Lasso — X List Assigner: Select tweets and assign their authors to your X Lists, in bulk, from the timeline." That is the entire public narrative. The manifest declares NO icons at all (no `icons` key, no `action.default_icon`) — the toolbar shows a generic placeholder and a store listing could not even render a tile. The PRD additionally promises a hover "+List" one-click quick action (user story 1, UX flow "Quick") that does not exist in the build.

2. INSTALL MOMENT — Chrome shows the host warning for x.com/twitter.com (static content script). The service worker's onInstalled handler is `console.debug("[Lasso] installed")` (src/background/index.ts) — no welcome tab, no explainer, no options page to land on. Nothing prepares the user for the permission warning or tells them what to do next.

3. FIRST RUN ON X.COM — The content script boots at document_idle, logs a colored console line, and (activation="auto" default) immediately mounts: a small circular checkmark button is prepended to EVERY tweet's author name (src/content/main.tsx injectOverlay). No tooltip beyond aria-labels, no coach mark, no explanation of why every tweet suddenly has a circle. The user is dropped cold.

4. FIRST SUCCESSFUL ASSIGN (AHA) — Click a checkmark → floating ActionBar appears bottom-center: "1 selected · [Add to list] · [Clear]". "Add to list" force-refreshes the list cache and opens the ListPicker (fuzzy input "Filter your lists…", usage-ranked, ↑/↓/Enter/Esc). Pick → human-paced run (700ms ± 30% jitter between adds) → Toast for a fixed 4s: "Added 1" / "Added 3 · 1 already in list". This loop genuinely works and is swift — but nothing in the product leads anyone to it.

5. LEARNING THE KEYBOARD LAYER — Alt+m (mute), Alt+n (not interested), Alt+l (add to list), x (toggle select), s (toggle select mode) are live immediately, targeting the hovered tweet or X's native j/k focus. There is no '?' help overlay, no hint chips, no keymap listing anywhere in the UI; the single piece of instructional copy — "hover a tweet (or press j to focus one) first" — is a console.warn invisible to users. Pressing `s` toggles a selectMode signal that NOTHING reads (grep confirms: written in selection-store/main.tsx, consumed nowhere) — visibly a no-op. Alt+m/Alt+n/block succeed silently and fail only to console.error.

6. DAILY HABIT / BULK SESSION — Hover+Alt+l is genuinely fast (auto-selects focused author + opens picker, possibly sweeping along an existing selection — documented open question). Bulk: check N tweets → "N selected" → picker → summary toast. ListUsage ranks frequent lists first — a nice invisible habit accelerator.

7. THINGS GOING WRONG — assign-to-list maps errors to outcomes: already-member (idempotent), protected ("N not allowed"), failed, rate-limited (STOP the run). BUT app.tsx pick() unconditionally runs selection.clear() — on a rate-limit stop the never-attempted remainder of the selection is destroyed, the toast says only "rate limit reached — stopped" (no count of skipped authors, no reset time though x-rate-limit-reset is documented as read). Logged-out / zero-lists / fetch-failure all collapse to `setLists([])` → an empty picker whose only copy is "No matching lists" — a wrong diagnosis with no escape hatch. If X redesigns and the selectors table goes stale, the scanner silently finds nothing; no user-facing health signal exists.

8. SETTINGS / DISCLOSURE — A typed settings model exists (backend "rest"|"dom"|"graphql", activation auto/on-demand, hotkeySelectMode, defaultListId in storage.sync) but there is NO settings surface: no options_ui, no popup, nothing. The GraphQL opt-in disclosure promised by PRD user story 5 and blueprint §7 cannot exist because there is no place to opt in. The code's actual default backend is "rest" (src/core/settings.ts) while PRD/CONTEXT/blueprint all tell a "DOM automation is the safe default" story — the documented trust narrative is stale. On-demand mode (ADR-0006) is reachable only by hand-editing storage; while inert there is zero page indication, the toolbar click gives no feedback, and the ADR's claim that "the select-mode hotkey" can wake it is false in the build (the keyboard layer is only installed inside start()).

9. TRUST AND UNINSTALL — Lasso's best trust asset — everything stays on-device, no servers, tokens never stored — is stated only in ADRs/docs, never to the user. List cache, usage-ranking data, and settings persist with no disclosure or wipe path. No uninstall URL, no feedback channel.

VOICE & NAME — Microcopy is consistently terse, sentence-case, "·"-separated ("N selected", "Add to list", "Added 3 · 1 already in list", "Filter your lists…", "No matching lists", "Nothing to add"). It is coherent but anonymous: the docs say "assign", the UI says "Add" (acceptable), and "not allowed" is the one vague phrase. The "Lasso" name appears to users only as the toolbar tooltip and a screen-reader-only aria-label ("Lasso selection"); the lasso metaphor — roping many authors in one throw, which is literally the product's core gesture — is used nowhere: no icon, no copy, no visual motif.

### Gaps

- **Discovery / store listing — No icon or brand asset anywhere — the Lasso metaphor is invisible:** src/manifest.config.ts declares no `icons` and no `action.default_icon`; the toolbar shows a generic placeholder and a Web Store listing has no tile. The lasso metaphor (rope many authors in one throw — a perfect match for multi-select) appears in zero pixels and zero user-visible copy except the toolbar tooltip and an aria-label. The redesign should make the rope/loop the visual identity: icon, the selection checkmark could be a loop, the ActionBar could read like a catch count.
- **Install moment / first-run — Zero first-run experience — user is dropped cold:** src/background/index.ts onInstalled is just console.debug. No welcome tab, no 60-second explainer of the three gestures (check → Add to list → done), no preparation for the 'read and change your data on x.com' permission warning, and no options page to land on. Then on x.com, checkmark circles silently appear on every tweet with no coach mark explaining what they are. Minimum fix: onInstalled opens a one-page welcome (or x.com with a one-time dismissible coach mark on the first overlay).
- **First-run / PRD drift — The promised hover '+List' quick action doesn't exist, and 'select mode' is a no-op:** PRD user story 1 and UX flow 'Quick' promise hover → +List → picker. The build's only mouse path is the always-visible checkmark + ActionBar (2 extra clicks). Meanwhile `s` / toggle-select-mode writes selection.selectMode, which nothing reads (grep: consumed nowhere in src/ui or src/content) — pressing s visibly does nothing, and PRD story 2's 'enter select mode' beat is dead. Either build the hover quick action and make overlays appear only in select mode (decluttering every tweet), or update the PRD and delete the dead command.
- **Learning the keyboard layer — No '?' help overlay, no hint chips — the keymap is undiscoverable; the only hint is a console.warn:** DEFAULT_KEYMAP (src/content/keyboard.ts) is documented only in docs/. There is no '?' cheat-sheet overlay (X itself sets the precedent with '?'), no shortcut hints in the ActionBar ('Add to list  Alt+L') or picker footer, no hint after the first mouse-driven assign ('Next time: Alt+L'). The one piece of guidance — '[Lasso] hover a tweet (or press j to focus one) first' in src/content/main.tsx:107 — is a console.warn no user will see; it should be an on-screen toast.
- **Daily quick-path habit — Quick actions (mute / not-interested / block) succeed and fail in total silence:** In src/content/main.tsx runCommand, Alt+m/Alt+n/block call REST or caret actions with no success feedback and console.error-only failure — directly violating the product's own FR7 'never silent; loud failures' ethos. Mute and block are consequential and need a confirming toast with an undo ('Muted @handle · Undo'); note the REST blockUser path also bypasses the always-confirm rule the keyboard blueprint promises for Block.
- **First bulk session / rate limit — Rate-limit stop destroys the un-attempted selection and hides the damage:** src/content/app.tsx pick() runs selection.clear() unconditionally after assignAuthorsToList; on a rate-limited STOP (src/core/actions/assign-to-list.ts breaks the loop), authors never attempted simply vanish — they're not in results, not counted in the toast ('rate limit reached — stopped' omits how many were skipped), and the user must re-find and re-select the tweets. Keep the remainder selected, and surface the reset time (the blueprint says x-rate-limit-reset is read) — 'Added 4 · rate limited — 6 still selected, try again in 12 min'.
- **Things going wrong — Logged-out, zero-lists, and fetch-failure all collapse into a misleading 'No matching lists':** src/content/app.tsx openPicker() catches every failure as setLists([]), and src/ui/ListPicker.tsx renders the same 'No matching lists' empty state. Three different stories need three different beats: logged out → 'Log in to X to see your Lists'; user has no Lists → 'You don't have any Lists yet' + a link to x.com list creation (the v1 non-goal of inline creation still deserves an escape hatch); transient failure → 'Couldn't load your Lists · Retry'.
- **Things going wrong — X redesign / selector breakage has no user-facing beat — Lasso just silently disappears:** CONTEXT.md designates content/selectors.ts as 'the one place to fix on an X redesign', but there is no detection or messaging: if article[data-testid=tweet] or the caret/dialog selectors go stale, the scanner injects nothing and DOM-driver errors ('Lasso: Add/remove from Lists menu item not found') die in the console. Add a lightweight health check (N mutations observed, zero tweets matched → one-time notice 'Lasso can't read the timeline — X may have changed; check for an update') so breakage looks like a known state, not a vanished product.
- **Settings / backend opt-in disclosure — No settings surface exists at all — the promised GraphQL disclosure is unreachable:** FR9 and user story 5 promise settings (backend strategy, default list, hotkeys, activation) with a clear policy disclosure on the GraphQL opt-in (blueprint §7: in-UI disclosure about private endpoints). The build has a full typed settings store (src/core/settings.ts) but no options_ui, no popup, nothing in the manifest — every setting, including the rebindable keymap and ADR-0006's activation toggle, is reachable only by hand-editing chrome.storage. The redesign needs an options page (or action popup) as a first-class chapter of the story.
- **Settings / trust narrative — Docs and build contradict each other on the safety default (DOM vs REST backend):** PRD, CONTEXT.md, and the blueprint all tell a 'sanctioned DOM automation is the safe default; GraphQL is opt-in with a caveat' trust story, but src/core/settings.ts ships DEFAULT_SETTINGS.backend = 'rest' — a third backend (RestXListApi) that the PRD story never mentions. Whatever disclosure copy eventually ships would be wrong today. Decide the real default, document REST's policy position, and realign PRD/CONTEXT/ADRs before writing user-facing trust copy.
- **Settings / on-demand mode — On-demand activation is half-built: no hotkey wake, no dormant indicator, no click feedback:** ADR-0006 says inert mode wakes via 'the toolbar action or the select-mode hotkey', but src/content/main.tsx installs the keyboard layer only inside start() — the hotkey cannot wake it; only the toolbar message works. While dormant there is zero indication anywhere that Lasso is installed-but-sleeping (no badge, no title change), and clicking the toolbar icon to wake it produces no visible acknowledgment. Add an action badge state (e.g. 'zz' / colored dot) and a wake toast, or fix the ADR.
- **Result feedback (aha moment durability) — The toast is a fixed 4-second whisper: failures evaporate with no detail, no retry, and overlapping runs clear each other:** src/content/app.tsx shows the summary for exactly 4000ms regardless of severity; 'protected'/'failed' outcomes never say WHICH author failed (AssignResult carries author + message, but src/ui/Toast.tsx renders only the one-liner), and there is no retry affordance. Also a real bug: the setTimeout is never cleared, so a second run's toast is wiped early by the first run's stale timer. Failures should persist until dismissed and expand to per-author detail.
- **Microcopy voice — Voice is consistent but anonymous, with two confusing phrases: 'not allowed' and author-vs-tweet counts:** The terse '·'-separated voice (src/core/result-summary.ts, ActionBar, ListPicker) is coherent and X-native — good foundation. But 'N not allowed' for protected accounts explains nothing (should be 'N protected — can't be added'); 'Nothing to add' reads like a shrug; and the ActionBar counts AUTHORS while checkmarks light per TWEET (selection keyed by screenName, so selecting two tweets by one author shows two lit checkmarks but '1 selected'). And no visible string ever says 'Lasso' — the name exists only in aria-labels and the toolbar tooltip, so the voice has no owner.
- **Trust and uninstall — The strongest trust facts — nothing leaves the device — are never told to the user, and there's no data/uninstall story:** ADR/blueprint invariants (no servers, tokens never stored, SW holds nothing, all calls same-origin in the user's own session) are Lasso's best retention/trust copy and appear nowhere user-facing. Meanwhile lasso:lists, lasso:list-usage, and lasso:settings persist in chrome.storage with no disclosure and no 'clear data' control, and there's no chrome.runtime.setUninstallURL / feedback beat. A one-paragraph privacy note on the (to-be-built) options page plus a wipe button closes the loop from install to uninstall.

Lasso's middle chapter — select, pick a list, get a summary toast — is genuinely built and swift, but it is a story with no beginning and no safety net. There is no install/first-run beat (onInstalled is a console.debug; checkmarks just appear on every tweet unexplained), no keyboard discoverability (no '?' overlay; the only instruction is a console.warn), no settings surface at all (making FR9 and the promised GraphQL policy disclosure unreachable), and error states collapse into one misleading 'No matching lists' empty picker. Several documented beats have drifted from the build: the hover '+List' quick action doesn't exist, 'select mode' (s) is a visible no-op, the code's default backend is 'rest' while every doc says DOM, and ADR-0006's hotkey-wake for on-demand mode isn't wired. The worst journey wound is the rate-limit path: the un-attempted selection is cleared and the skipped count hidden. Voice is consistent and X-native but anonymous — the Lasso name/metaphor appears in no icon and no visible copy, and the extension ships with no icon at all. 14 concrete gaps reported across discovery, install, first-run, keyboard learning, daily quick-path, bulk, error states, settings/disclosure, and trust/uninstall.

## Inspiration research

Research distilled from Raycast/cmdk, Linear, Superhuman, Vimium, Arc, Sonner/Emil Kowalski, Grammarly engineering, and X's own design language, aimed at redesigning Lasso's five surfaces (C:/Users/fenchem/xtimelinefilter/src/ui/ListPicker.tsx, ActionBar.tsx, Toast.tsx, TweetOverlay.tsx, styles.css) plus the missing product story (onboarding, discoverability, errors).

(1) Palette anatomy is the biggest gap vs. best-in-class. The cmdk Raycast theme gives an exact spec: 12px container radius with 8px vertical padding; 15px input at 8x16 padding; 40px result rows at 14px with an 18px leading icon slot, 8px row radius, 8px gap; 12px muted group headings; a 40px footer separated by a 1px hairline; 20px keycap chips at 4px radius with shortcuts right-aligned. ListPicker.tsx today renders name-only 15px rows with no icon, no metadata (member count/privacy), no section headers (Recent vs All lists), no footer hints, and a single-line empty state — those five slots are exactly where the redesign should land.

(2) Keyboard-first products teach in-product, never in docs: Superhuman/Linear echo the direct shortcut on every palette row until the user graduates past the palette; Raycast's footer bar exists explicitly to teach shortcuts; Vimium's '?' opens a live in-page cheat sheet; Superhuman ships labels that decay after 7 days (added after 15x15px icons measurably blocked activation; onboarding work yielded +20% shortcut usage, +67% feature adoption). This composes into Lasso's onboarding stack at near-zero UI cost: footer kbd bar + shortcut-echo chips + decaying first-run coachmarks + a '?' overlay inside the Alt layer.

(3) Speed and error story: Superhuman's 50-60ms internal response target plus optimistic UI (act instantly, reconcile in background, undo toast with its own shortcut) defines the 'even swifter' UX — add-to-list should flip checkmarks immediately and treat the API as eventually consistent, with failures arriving as corrective toasts, never blocking dialogs. Emil Kowalski's rules (already cited in the code) get sharper edges: never animate keyboard-initiated UI, <300ms, transform/opacity only, interruptible transitions — while Sonner's deliberate 400ms 'ease' shows system-initiated toasts are the sanctioned exception. Toast.tsx should adopt Sonner's full mechanics (stack scale 0.05*index, 14px gap, 4s auto-dismiss paused on hidden tab, hover-expand, velocity-based swipe dismiss); X itself ships Sonner, so that feel reads literally native.

(4) Visual language: Raycast and Linear both build depth from a 3-4 step surface ladder with 1px hairlines instead of shadows, confine accent color to focus rings and the single primary CTA, and run dense 12-14px type with 12-13px section headers — styles.css currently has only a 2-step ladder and one heavy 24px shadow. To blend with X specifically: prepend 'TwitterChirp' to the font stack, keep #1d9bf0-adjacent accent and 9999px pills, use 16px dialog radius and 15px/20px body type; note X's post-2021 direction is fewer dividers, less blue, more whitespace — restraint reads as native. Grammarly's engineering doctrine covers coexistence for tweet-scanner.ts/TweetOverlay: render adjacent to (never inside) host DOM, position via getClientRects, prefer IntersectionObserver over MutationObserver polling (continuous getBoundingClientRect can eat 90% CPU), and harden the Shadow DOM with :host{all:initial} because host custom properties pierce the boundary. Arc's Little Arc/Peek contributes the overlay grammar — every surface ephemeral, consistent Esc/outside-click dismissal, exactly one promote action — and Vimium's hint mode suggests the one genuinely new interaction worth prototyping: lettering visible tweets for direct multi-select instead of j/k walking.

## Design directions considered

### Native Chameleon — "Shipped by X"

*Indistinguishable from X. Faster than X.*

Tradeoffs: 1. CHAMELEON FRAGILITY: every token is pegged to X's current visual language. When X rebrands (it has twice in four years), Lasso looks instantly stale until a token pass — worse than a neutral design would. Mitigation is cheap (all values live in one @theme block, hexes documented inline) but it is a standing maintenance tax, and the theme detector adds one more X-coupling beyond selectors.ts. 2. ZERO BRAND EQUITY: success means nobody ever notices Lasso. No screenshot is shareable, 'what extension is that?' never happens, and word-of-mouth growth is structurally capped. The icon and the two tiny wordmarks are the entire brand. If distribution ever matters, this lens is the wrong one. 3. ORIGIN-CONFUSION RISK: deliberately imitating X's dialogs flirts with passing-off; a user may blame X for Lasso's bugs (or X may object). We never use X's logo or name in UI, and settings/welcome state plainly that Lasso is a third-party extension — but the ambiguity is inherent to the lens. 4. ONE KNOWING AA VIOLATION: the filled 'Add to List' CTA keeps white-on-#1d9bf0 at 15px/700 (~2.9:1) because a darker blue button would be the single most visible tell that this isn't X. Scope is contained — every informational string is ink-on-surface at 7:1+, toasts moved off blue entirely, and --color-accent-text exists for all text-on-surface blue. A 'higher-contrast buttons' toggle in settings is the honest escape hatch if this bites. 5. ANCHORED PICKER COSTS: caret-anchoring needs flip logic, viewport clamping, an rAF reposition loop while open, and a freeze fallback when X's virtualized timeline recycles the anchor mid-open — roughly 80 lines that bottom-center never needed, and the freeze case will occasionally look detached. The perceived-nativeness payoff justifies it, but it is the riskiest code in the redesign. 6. HIDDEN-BY-DEFAULT OVERLAYS trade timeline cleanliness for serendipity: a user who skips the welcome card may never discover selection (the always-on circles, ugly as they were, advertised the feature on every tweet). The decaying hints and the '?' sheet are the counterweight; watch activation rate after shipping and re-add a subtle permanent affordance if it craters. 7. RESTRAINT READS AS UNDESIGNED to the owner who asked for a redesign: square full-bleed rows, no surface ladder, no keycap gradients, no entrance animations — the Raycast/Linear polish vocabulary is deliberately forbidden here. The wins are felt (speed, anchoring, states), not seen in a side-by-side. 8. PREREQUISITE DEBT: this direction surfaces decisions code must settle first — the rest-vs-DOM default contradiction between src/core/settings.ts and PRD/CONTEXT must be resolved before the settings disclosure copy can ship, and select-mode plumbing (selection.selectMode consumers) must land before the mode indicator means anything. 9. OKLCH approximations of X's hexes were derived analytically, not measured; budget a one-hour pass comparing rendered Lasso surfaces against live computed styles on all three X themes before calling the camouflage done.

### Command Layer

*A Raycast for your timeline: Lasso stops hiding inside X and becomes the glassy command deck floating above it.*

Tradeoffs: 1) DISTINCTNESS CONTRADICTS THE CODEBASE'S STATED PHILOSOPHY. styles.css and the PRD say \"blend into X\"; this direction deliberately breaks that. A glassy dark Raycast panel on light-mode X is a statement some users will read as foreign chrome on their feed. The competing \"camouflage\" direction would be safer for mass appeal; Command Layer optimizes for the power-user niche who buys Superhuman. If the owner wants invisible, this is the wrong lens. 2) GLASS COSTS FRAMES. backdrop-filter: blur(20px) over a virtualized, image-heavy timeline forces compositing work; on low-end Windows laptops it can jank scroll while the ActionBar is visible. Mitigations are specced (@supports fallback to opaque surface-1, blur only on visible panels, scrim without blur) but the honest floor is: opaque panels are cheaper and we may need a \"reduce transparency\" setting. 3) SCOPE IS LARGE. This is not a reskin: three-theme detection, facepile + review popover, progress-preserving assign flow, Sonner-grade toast deck, mode chip, coachmarks with decay state, '?' overlay, an entire options page, icons, and undo (which requires remove-member API support that may not exist in the DOM backend — if it doesn't, Undo ships API-backend-only or not at all). Realistic sequencing: palette + tokens + toast variants first (the three audit pillars), onboarding/settings second, undo last. 4) KEYCAP DENSITY HAS A CEILING. Chips on every button serve learners and experts but add visual noise; on a 360px-wide viewport the ActionBar with facepile + count + labeled button + keycap will overflow — the spec's mobile fallback (drop facepile to 2, hide keycap) erodes the very identity being sold. 5) CENTERED PALETTE COVERS CONTENT. At 20vh the panel sits over tweets, unlike the old bottom placement; for a hover-targeted Alt+L flow there is a real argument for anchoring near the focused tweet instead — we chose genre legibility and a never-moving input over proximity, and that is a taste call the owner should ratify. 6) THEME DETECTION IS A SELECTOR-SHAPED LIABILITY. Reading body background to infer Default/Dim/Lights-Out is exactly the kind of X-redesign-fragile coupling CONTEXT.md warns about; it needs the same health-check treatment as the tweet scanner. 7) 'Z' UNDO AND '?' EAT MORE OF THE PAGE'S KEYSPACE. Every new unmodified binding risks colliding with X's native shortcuts (X already uses '?' for its own help on some surfaces) — the layer must keep deferring to typing targets and may need to scope 'Z' to the 10s post-toast window only."}

### Roped In

*Catch them as you scroll — one throw adds them all.*

Tradeoffs: 1. THE AMBER BREAKS CAMOUFLAGE — DELIBERATELY. The research says "quieter is more native," and this direction spends that nativeness on memorability. Risk: warm amber near X's blue links/likes can vibrate. Mitigation: cold-neutral surfaces everywhere, amber confined to Lasso-owned controls and exactly one in-timeline pixel-group (the avatar-corner loop); never amber text inside X's own layout. If the owner ultimately wants invisibility over identity, this whole direction is the wrong pick — that's the honest fork.  2. METAPHOR COPY HAS A CRINGE CEILING AND A LOCALIZATION COST. "Roped 3 into Design Heroes" is the love-it moment; "X pulled the reins" is one notch from twee. The six-word glossary cap and the "plain verbs for errors" rule are the guardrails, but every future contributor will be tempted to add cowboy puns — the voice doc must be enforced in review. Localizing "rope/throw/reins" idiomatically is genuinely hard; if i18n is ever on the roadmap, the playful titles need a parallel literal string table.  3. AVATAR-CORNER OVERLAY IS THE RISKIEST ENGINEERING CHANGE. Absolutely positioning over X's avatar block depends on X's container being position:relative and survives less DOM churn than the current inline prepend. The spec carries a fallback (in-name-row placement, select-mode-gated), but expect this to be the component that breaks on an X redesign. The payoff — zero layout shift on every tweet — is worth the fragility budget.  4. TOP-ANCHORED PALETTE TRADES PROXIMITY FOR STABILITY. Alt+L from a focused tweet now opens UI ~400px from the user's gaze instead of near the tweet. Per-tweet anchoring was rejected because it fights virtualized scrolling and makes the surface jump between invocations; a fixed home builds muscle memory. Keyboard users pay a small eye-travel tax.  5. NOT FULLY OPTIMISTIC. Superhuman-style optimism would flip "added" state instantly; list membership has no cheap, reliable rollback across three backends (especially the DOM driver), so the design keeps honest progress UI ("Roping… 2/5") instead of lying about server state. Perceived speed is bought elsewhere: cache-first instant palette open, instant loop toggles, 0ms keyboard surfaces. Slightly less magical, never wrong.  6. MORE SURFACE AREA, MORE STATE. Welcome card, coachmarks with decay counters, select-mode chip, help overlay, options page, theme observer, selector health check — roughly 2-3x the current UI code, plus storage keys (lasso:onboarded, lasso:coach:*) that need the settings wipe button to stay truthful. The audit's worst wounds (clear-before-resolve, force-fetch, fake empty state) are cheap fixes; the brand layer is where the budget goes.  7. UNDO IS A PROMISE WITH EDGES. Toast "Undo" issues real member-remove calls; if the user was ALREADY in the list before Lasso added them (already-member outcome), undo must skip them or it silently removes a pre-existing membership. The spec scopes undo to outcome==='added' results only — easy to get wrong in implementation, called out here so it doesn't ship as a data-loss bug.  8. OPEN DOC/CODE CONTRADICTION BLOCKS ONE COMPONENT. The settings page's backend-disclosure copy assumes DOM automation is the default, but src/core/settings.ts ships backend:'rest'. That product decision (and PRD/CONTEXT/ADR realignment) must land before the trust copy does — shipping the copy against the current default would be lying in the exact place trust is being built.

## Judge verdicts

### Verdict — winner: Native Chameleon — "Shipped by X"

- Native Chameleon — "Shipped by X": **9/10** — The most uncompromising speed posture of the three: 0ms motion on BOTH picker open paths (keyboard and pointer), cache-first instant open, optimistic check flips, progress that never destroys selection, and no backdrop-blur frame tax anywhere. The caret-anchored picker is the killer keyboard move — Alt+L opens the palette where your eyes already are (at the j/k-focused tweet), eliminating the 400px gaze hop the other two impose, and the anti-jump body-height rules mean the input never moves under a fast typist's caret. Discoverability rides X's own conventions ('?' is X's shortcuts key — the entire audience already knows it) plus a permanent footer kbd legend and decaying chips. The one real risk to speed is the rAF anchor-tracking loop and freeze fallback — ~80 lines of the riskiest code in any direction — but it buys gaze proximity, which compounds on every single assign.
- Command Layer: **7/10** — Best-in-class shortcut discoverability — keycap chips on every actionable element, platform-aware Alt/⌥ rendering, live-keymap '?' sheet, fixed-home palette for spatial muscle memory — and the Esc priority grammar (palette > popover > select mode > clear) is the cleanest keyboard semantics anywhere in the three. But backdrop-filter: blur(20px) on a persistent ActionBar over a virtualized image-heavy timeline is a standing threat to scroll frames, which is the cardinal sin for a speed purist; the spec itself admits opaque is cheaper. The top-20vh palette also costs eye travel from the focused tweet on every Alt+L, and the glassy dark-first identity buys zero milliseconds. Strong mechanics wearing an expensive coat.
- Roped In: **8/10** — Contains the single best keystroke-economy idea in the whole exercise: ⌥⇧L throws straight to a Default List, skipping the picker entirely — for the heavy repeat user that collapses the whole flow to one chord, sub-second. Also the only direction with a keyboard-reachable error recovery ('r' retries in the picker error state) and well-calibrated coachmark decay (max 2 graduation hints, keyboard assigns never nagged). Costs: the fixed top palette pays the same eye-travel tax as Command Layer, the click-catcher sacrifices feed scrolling while the palette is open (Chameleon explicitly never blocks scroll), and rope vocabulary ('3 roped', 'X pulled the reins') adds a tiny parse cost at exactly the moments a speed reader wants instant scan — charm a purist would trade away.

Graft-worthy ideas: Graft these into the Chameleon build regardless:  FROM ROPED IN: 1. ⌥⇧L / Alt+Shift+L direct-throw to the Default List, bypassing the picker entirely — the biggest time-to-assign win available; Chameleon specs a Default List setting but never gives it a chord. This must ship. 2. 'r' retries while the picker error state is shown — keyboard-reachable recovery, no mouse reach for the Retry pill. 3. 'Stop' affordance during the assign run (aborts remaining sequential calls, keeps un-attempted authors selected) — Chameleon has retry-after-failure but no mid-run abort. 4. The Undo scoping rule from tradeoff 7: Undo applies ONLY to outcome==='added' results, never pre-existing memberships — prevents a silent data-loss bug whoever implements Undo. 5. The concrete selector-health heuristic (>200 mutations with 0 tweet matches → one-time 'X may have changed' toast) — Chameleon mentions the toast but not the trigger. 6. Toolbar badge showing the live roped/selected count — free ambient state at zero pixels in-feed.  FROM COMMAND LAYER: 7. The explicit Escape priority grammar: palette > review popover > select mode > clear selection — deterministic Esc semantics; Chameleon handles Esc per-surface but never defines the stack order. 8.
