# Convex is an optional off-to-the-side membership mirror — never the source of truth

Lasso has been deliberately backend-free: authenticated x.com calls run in the content script, the SW holds no tokens and no long-lived state, and a List's membership lives only on X. We want three things that a stateless extension can't give us: a durable **log** of every add/remove, "already in" marks that are **instant and survive across devices**, and a view of List membership **across the several X accounts** one person operates. We add **Convex** to get them — but as a **Mirror**, not a brain.

The load-bearing decision: **X stays the source of truth; the Mirror only records what we did against X and caches X's answers.** The Mirror never drives X (no reconciler pushing Convex state back to X), and a Mirror failure never blocks or alters the X flow — the assign run, its toast, and undo proceed exactly as today whether or not the write to Convex succeeds. This keeps the existing `XListApi` path untouched and makes the whole feature **additive and optional**: with no device key configured the Mirror is absent (`NullMembershipStore`) and the extension behaves bit-for-bit as before.

Concretely:

- **A new seam, sibling to `XListApi`.** `MembershipStore` has a `ConvexMembershipStore` and a `NullMembershipStore`, chosen by a factory from settings and pinned by a shared contract test — the same shape the backends already use. Consumers (controller, picker) depend only on the interface.
- **Owner captured at action time.** When you operate on a List, we read the account logged into x.com *right then* (numeric `userId` from the `twid` cookie, screenName best-effort) and stamp it as the **Owner** on the record. No background polling, no account registry — it rides the existing "one explicit gesture → one run" invariant. ("Account" stays reserved for the *member*; the operator is the **Owner**.)
- **Reconcile is per-author and lazy.** On picker open for one author we mirror X's `memberships.json` answer ("which of my Lists contain this person") into the snapshot — cheap, maps 1:1 onto the existing membership check. We do **not** pull full per-List rosters (paginated, rate-limit-prone, and only needed for a member-browsing screen we aren't building). The snapshot is therefore a lazily-filled (List, screenName) presence map: complete for anyone you've actually checked, which is exactly what "already in" needs.
- **Cross-account is read-only off-session.** The picker groups Lists by Owner from the Convex catalog, but you cannot write to account A's List while logged in as B (X rejects it cross-session). So only the **active Owner**'s Lists are pickable; the rest are shown disabled with a "Switch to @owner" hint, and their "already in" marks come from cached Convex data labelled "as of last use" (we can only freshly reconcile the account currently logged in).
- **The log keeps every outcome.** Each attempt — `added | already-member | protected | rate-limited | failed`, add or remove — appends one **Audit event**. The **snapshot** changes only on a real membership change.

This amends one invariant: the extension now holds **exactly one** long-lived credential, the Mirror **device key**, validated by every Convex function (single-tenant, personal store; the deployment is yours). X session tokens remain ephemeral and content-script-only — the device key is unrelated to X auth and grants nothing on X.

Trade-offs accepted. The snapshot can lag X for members added directly on x.com until the next per-author reconcile touches them — acceptable because X's own check still runs for the active Owner and corrects the snapshot, and "already in" is a hint, not a gate. Cross-account checks for a non-active Owner are inherently stale (we can't query X for an account we're not logged into), which is why they're labelled, not trusted. And the device key lives in the content script's isolated world rather than the SW — chosen so the reactive WebSocket survives (MV3 service workers are killed aggressively, which would break the live "already in" subscription the SW-broker alternative depended on).

Tests must encode "Mirror is optional and never load-bearing": a controller test proves a thrown/rejecting `ConvexMembershipStore` leaves the assign toast, undo, and selection identical to the `NullMembershipStore` run. Convex functions are covered with `convex-test` (recordAssign upserts Owner+List and appends an event; reconcile mirrors `memberships.json` truth including removals; `listsContaining` is reactively correct; a wrong device key is rejected). And `getCurrentAccount` must be **live-verified** against x.com, not just fixtured — reading the wrong account would silently mis-attribute every record, the same fire-and-forget trap ADR-0008 was written to kill.
